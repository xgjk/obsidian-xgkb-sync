# Obsidian 与玄关知识库同步方案研究

**——双中台架构下的个人知识库同步插件设计困境与出路**

> 作者：Codex（AODW_CODEX）
> 日期：2026-05-02
> 状态：研究论文 v1.0

---

## 摘要

本文档记录了为 Evan 设计并研究 Obsidian 与玄关知识库双向同步插件的全过程。研究过程中，我们先后接触了两个定位截然不同的后端系统：**玄关知识库（XGKB）**——一个有成熟管理界面和权限体系的个人知识库平台；以及**XG-File 文件服务**——一个纯粹的分布式文件存储中台，提供 WebDAV 协议支持但缺少管理界面。通过对两个系统的深入分析，我们发现最初设想的"同步插件"问题，本质上是一个**双中台架构选择**问题：到底是复用现有知识库平台的管理能力，还是迁移到新文件服务的灵活性？本文详细记录了这个判断过程中的技术发现、核心矛盾和未来可能的方向。

---

## 一、研究背景与目标

### 1.1 原始需求

Evan 的原始需求是：**开发一个 Obsidian 插件，实现 Obsidian 与"玄关知识库"之间的双向同步**。

背景信息：
- Obsidian 是一个本地优先的个人知识管理工具，笔记以 Markdown（.md）文件形式存储在本地 Vault 目录中
- Evan 希望将 Obsidian 作为主要写作工具，同时利用玄关知识库的平台能力（管理、权限、分享）对外提供知识服务
- 多设备场景：Mac/iPhone/iPad 通过 iCloud 同步同一 Obsidian Vault，云端统一为玄关知识库

### 1.2 约束条件（项目边界）

基于 RT-001 v2.0 规格，我们为 MVP 设定了以下约束：

| 约束 | 说明 |
|------|------|
| 只同步 .md 文件 | 图片/附件不纳入同步范围 |
| 手动触发同步 | 无定时自动同步 |
| 冲突处理：Last-Write-Wins | 以 mtime 为准，不保留冲突副本 |
| 增量同步 | 基于 SyncStateDb 记录，上下文感知 |
| 只支持个人空间 | 不涉及项目空间 |
| 不保留版本历史 | 同步覆盖，不保留历史副本 |

### 1.3 研究方法

本研究采用**第一性原理**方法：
1. 追溯原始需求的核心目的（Why）而非直接跳入技术方案（How）
2. 对已发现的双系统分别进行接口能力和产品定位分析
3. 以"功能-成本"矩阵评估各方案可行性
4. 不预设结论，让证据驱动决策

## 二、系统发现与分析

### 2.1 系统一：玄关知识库（XGKB）

**定位**：成熟的个人知识库管理平台

**认证方式**：`Header appKey`

**API 根地址**：`https://sg-al-cwork-web.mediportal.com.cn/open-api/`

**核心能力（基于 RT-001 调研）**：

| 能力 | 支持情况 |
|------|---------|
| 文件上传（uploadContent） | ✅ 支持，可传 .md 内容 |
| 目录遍历（getChildFiles） | ✅ 支持，递归获取文件树 |
| 文件下载（getDownloadInfo） | ✅ 支持，获取下载链接 |
| 文件夹创建 | ⚠️ 无直接 API，需用文件占位 |
| 权限管理 | ✅ 管理界面支持 |
| 分享能力 | ✅ 管理界面支持 |
| 版本历史 | ❌ API 未暴露 |
| WebDAV | ❌ 不支持 |

**结论**：XGKB 是一个**内容平台**，有完整的知识库管理界面（权限、分享），但其开放 API 设计为**辅助性质**，主要用于应用集成，接口粒度较粗，缺少文件系统的树形操作原语（如 mkdir、rmdir）。

### 2.2 系统二：XG-File 文件服务

**定位**：分布式文件管理中台

**认证方式**：`Header appKey`

**API 根地址**：`https://xg-file.mediportal.com.cn`

**核心能力（基于 API 文档 v1.2）**：

| 能力 | 支持情况 |
|------|---------|
| 业务桶管理（Bucket） | ✅ 多租户隔离 |
| 文件上传（单文件直传） | ✅ multipart，支持物理去重（MD5） |
| 目录列表（list） | ✅ 支持递归 |
| 全路径模糊搜索 | ✅ keyword 搜索 |
| 历史版本管理 | ✅ 覆盖上传保留版本 |
| WebDAV 协议 | ✅ 原生支持，标准协议 |
| 管理界面 | ❌ 无，需自建 |
| 权限管理 | ⚠️ 桶级别隔离，无细粒度权限 API |
| 分享能力 | ❌ 无原生分享接口 |

**WebDAV 特别说明**：

XG-File 在 v1.2 文档中新增了完整的 WebDAV 支持，这是一个重要发现：
- 挂载地址：`https://xg-file.mediportal.com.cn/dav/{bucketCode}/`
- 认证：任意用户名 + `appKey`（Basic Auth）
- 文档甚至明确推荐 Obsidian 用户通过 **Remotely Save 插件**的 WebDAV 模式实现零开发同步

这意味着，如果我们选择 XG-File 作为存储后端，Obsidian 端的同步**可以完全不写代码**——只需配置 Remotely Save 插件即可。

## 三、核心矛盾：存储 vs. 管理

### 3.1 两个系统本质上是互补的

将两个系统并列对比，我们会发现一个明显的互补关系：

| 维度 | XGKB（知识库） | XG-File（文件服务） |
|------|---------------|-------------------|
| **产品定位** | 内容管理平台 | 分布式存储基础设施 |
| **管理界面** | ✅ 成熟完整 | ❌ 完全缺失 |
| **权限体系** | ✅ 完善 | ⚠️ 仅桶级别隔离 |
| **分享能力** | ✅ 有 | ❌ 无 |
| **API 粒度** | 粗（内容级） | 细（文件系统级） |
| **WebDAV** | ❌ 不支持 | ✅ 原生支持 |
| **物理去重** | ❌ 未提及 | ✅ MD5 级别 |
| **版本管理** | ❌ 不透明 | ✅ API 可查 |

**XGKB 的优势恰好是 XG-File 的短板，反之亦然。**

### 3.2 Evan 提出的关键洞察

在研究过程中，Evan 提出了一个关键的洞察：

> "我们需要的不是一个简简单单的存储，我们需要这个存储也要有一套管理界面。现有的玄关知识库本身已经有了一套成熟的管理界面，可以做权限管理和分享。新提供的文件中台似乎缺少这一块管理界面，只是一个文件存储服务。"

这个观察直击问题的核心。我们最初把"同步插件"理解为一个纯粹的技术问题（如何搬运文件），但 Evan 帮助我们看到，这实际上是一个**产品架构问题**：

- **如果选 XGKB**：同步的技术成本低（已有 API），但需要为 Obsidian 插件额外解决 API 粗糙带来的体验问题（如文件夹创建）
- **如果选 XG-File**：存储体验好（WebDAV、增量同步、版本管理），但失去了管理界面和权限分享能力
- **理想情况**：希望同时拥有 XGKB 的管理能力和 XG-File 的存储体验

### 3.3 两种架构路径

**路径 A：基于 XGKB 构建（当前 RT-001 方向）**

```
Obsidian Vault ←→ [Sync Plugin] ←→ XGKB API ←→ XGKB 管理界面
```

优点：
- 直接复用现有管理界面（权限、分享）
- API 已可用，研究成本低

缺点：
- XGKB API 粒度较粗，文件夹创建需要 workaround
- 同步逻辑全权由插件负责，版本管理能力弱
- 多设备冲突处理受限

**路径 B：基于 XG-File 构建**

```
Obsidian Vault ←→ Remotely Save (WebDAV) ←→ XG-File ←→ [需自建管理界面]
```

优点：
- WebDAV 协议原生支持，Obsidian 端零开发
- 存储层能力完整（去重、版本、WebDAV 锁）
- 逻辑清晰，存储与管理分离

缺点：
- **完全缺失管理界面**——这是 Evan 指出的核心问题
- 权限、分享需要从零建设
- 如果 XGKB 和 XG-File 数据不同步，则变成了两套系统

**路径 C：双中台协同（未来方向）**

```
Obsidian Vault ←→ XG-File (存储层) ←→ [管理界面开发] ←→ XGKB (管理层)
```

这是最理想的架构，但需要确认：XG-File 是否有计划提供管理界面？或者需要我们自建？

## 四、技术研究与发现

### 4.1 同步算法设计（RT-001 v2.0）

我们在 RT-001 中设计了一套基于 **Last-Write-Wins（LWW）** 的双向同步算法：

**核心规则**：以 mtime（文件修改时间）为唯一权威，本地时间和云端时间对比，较新者覆盖较旧者。

**决策矩阵**（有 record 的增量同步场景）：

| 本地变化 | 云端变化 | 动作 |
|---------|---------|------|
| ❌ | ❌ | 跳过 |
| ✅ | ❌ | 上传 |
| ❌ | ✅ | 下载 |
| ✅ | ✅ | 对比 mtime，较新者胜出 |

**数据结构**：SyncStateDb（IndexedDB）记录每个文件路径的 localMtime、remoteMtime 和 xgkbFileId，实现增量同步。

### 4.2 XGKB API 的粗糙性带来的挑战

RT-001 研究过程中，我们发现 XGKB API 存在以下限制：

1. **无直接文件夹创建 API**：需要通过在父目录创建空文件来"占位"目录，或者依赖 uploadContent 的 folderName 参数间接创建
2. **API 粒度以内容为中心**：getChildFiles 返回的是文件列表，没有原生的目录树操作（mkdir/rmdir）
3. **mtime 的语义**：云端的 mtime 是服务器时间（北京时区），需要保证本地时钟同步

### 4.3 XG-File WebDAV 的潜力

v1.2 文档中 XG-File 的 WebDAV 支持是一个意外收获：

- Obsidian 的 Remotely Save 插件本身支持 WebDAV
- 这意味着如果我们选 XG-File，**Obsidian 端可以零开发**
- WebDAV 支持 Exclusive 写锁（Redis 实现），可防止并发编辑覆盖

但这并不能解决 XG-File 缺少管理界面的问题。

## 五、待解决的核心问题

### 5.1 数据层问题（需要确认）

> **XGKB 的内容和 XG-File 的内容是什么关系？**

这是整个项目的先决问题：
- 如果 XGKB 底层就是用 XG-File 存储，那么两条路径最终指向同一份数据，可以组合使用
- 如果 XGKB 和 XG-File 是独立的两套存储，插件选任何一个都可以，但无法同时利用两者的优势

### 5.2 管理界面问题（需要决策）

> **XG-File 的管理界面缺失，是接受现状还是从零建设？**

可能的答案：
1. **接受现状**：纯靠 WebDAV + 文件操作，放弃细粒度权限和分享
2. **等待官方**：向 XG-File 团队提出管理界面需求
3. **自建管理界面**：作为插件的一部分或独立服务提供

### 5.3 架构方向问题（需要决策）

> **插件的后端选 XGKB 还是 XG-File？**

| 方案 | 管理界面 | 存储体验 | 开发成本 |
|------|---------|---------|---------|
| XGKB 直连 | 复用 ✅ | 一般 ⚠️ | 中等 |
| XG-File + 自建管理 | 需从零建设 ❌ | 好 ✅ | 高 |
| XG-File + 接受无管理 | 无 ❌ | 好 ✅ | 低 |
| XG-File + XGKB 协同 | 部分复用 ⚠️ | 好 ✅ | 待定 |

## 六、未来可能的方向

### 6.1 短期路径：基于 XGKB 完成 MVP

继续推进 RT-001，使用 XGKB 作为同步后端，尽快产出可用版本。

- 优点：管理界面现成，研究成本最低
- 缺点：API 粗糙，同步体验受限

### 6.2 中期路径：引入 XG-File 作为存储优化

如果 XG-File 与 XGKB 底层数据互通，可以考虑：
- Obsidian 端通过 WebDAV 直连 XG-File（Remotely Save 插件，零开发）
- 管理界面继续使用 XGKB
- 实现存储与管理分离的最优架构

### 6.3 长期路径：自建统一管理层

如果 XG-File 确认无管理界面开发计划，且双中台数据不同步：
- 基于 XG-File 的 REST/WebDAV API，从零构建管理界面
- 将权限管理、分享、审计等能力补齐
- 这本质上是一个**独立产品**的开发工作，不只是插件开发

### 6.4 研究方向：多中台协同架构

在企业级应用中，多个存储/管理中台并存的场景非常普遍。一个有价值的研究方向是：**设计一套通用的"同步中间层"抽象**，可以连接任意存储后端（XGKB、XG-File、S3、WebDAV）和任意管理层（自建、第三方），让上层应用（Obsidian 插件）不感知底层差异。

---

## 七、结论

本研究的核心发现是：**Obsidian 同步插件的选型，本质上是双中台架构选择问题，而非单纯的技术实现问题。**

玄关知识库（XGKB）和 XG-File 文件服务代表了两种不同的设计哲学：
- **XGKB**：以管理为中心，存储为管理服务（内容平台思维）
- **XG-File**：以存储为中心，管理界面留给客户自建（基础设施思维）

Evan 的洞察——"我们需要管理界面，而不只是存储"——为我们指明了方向。下一步需要与产品侧确认：XG-File 是否有管理界面的 roadmap？双中台之间的数据关系是什么？这些问题的答案将决定插件的最终架构。
